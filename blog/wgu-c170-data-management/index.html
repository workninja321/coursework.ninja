WGU C170 Data Management: SQL and Database Fundamentals Guide

December 21, 2025 8 min read Course Help

WGU C170 (Data Management - Applications) is the SQL-heavy course that turns database theory into hands-on query and design skills. If you can write clean SQL, normalize a schema, and explain why your design choices match the business rules, you are in great shape. This guide covers the official course focus, the exact skills you need, and a practical study plan to reach competency efficiently.

Official course focus (WGU catalog)

The WGU Institutional Catalog describes Data Management - Applications as a course that develops SQL expertise with MySQL and requires students to create and modify databases, tables, views, keys, and indexes. It also emphasizes SQL queries that range from simple single-table statements to complex multi-table joins. The catalog lists Data Management - Foundations as the prerequisite.

  • Use MySQL and SQL syntax to analyze data
  • Create and modify databases, tables, views, keys, and indexes
  • Write queries from basic single-table to complex multi-table joins
  • Complete prerequisite: Data Management - Foundations

Skills you must demonstrate

Think of C170 as two parallel tracks: query fluency and schema quality. You should be able to:

  • Write correct, readable SQL under time pressure
  • Model entities and relationships with clean keys
  • Normalize tables to reduce redundancy
  • Translate business requirements into query logic

SQL fundamentals to master

These are the core building blocks you will use every week in C170:

  • SELECT: filtering with WHERE, sorting with ORDER BY, and limiting results
  • Aggregations: COUNT, SUM, AVG with GROUP BY and HAVING
  • JOINs: INNER, LEFT, and RIGHT joins across multiple tables
  • Subqueries: filtering or comparing with nested SELECT statements
  • DDL: CREATE, ALTER, DROP
  • DML: INSERT, UPDATE, DELETE

Core database terms you should know

  • Entity: A real-world object (student, course, instructor) represented as a table
  • Attribute: A property of the entity (name, email, credits)
  • Primary key: A unique identifier for each row
  • Foreign key: A field that links one table to another
  • Cardinality: How many rows in one table relate to rows in another
  • Referential integrity: The rule that foreign keys must match existing primary keys
  • View: A saved query that can be reused
  • Index: A structure that speeds up lookups

SQL patterns you should recognize

Most exam and performance tasks are variations of a few patterns. Practice these until they feel automatic:

  • Multi-table joins with filtering and sorting
  • Aggregations by category (SUM and COUNT grouped by a dimension)
  • Subqueries that filter based on a calculated value
SELECT s.student_id, s.name, COUNT(e.course_id) AS courses
FROM students s
JOIN enrollments e ON s.student_id = e.student_id
GROUP BY s.student_id, s.name
HAVING COUNT(e.course_id) > 3
ORDER BY courses DESC;

If you can build queries like this without looking up syntax, you are well on your way to passing.

Data modeling and normalization

Schema design is where many students lose points. If you treat the database as a single table, you will struggle. Instead, focus on relationship clarity and clean normalization:

  • Identify entities, attributes, and relationships
  • Assign primary keys and enforce foreign keys
  • Normalize to 3NF to remove partial and transitive dependencies
  • Check cardinality and optionality in ER diagrams
Common pitfall: skipping normalization checks leads to duplicated data and broken joins.

Normalization quick walkthrough

Suppose you start with a single table that stores student info and course info together. That design creates repeated student data every time a student enrolls in a new course. In a normalized design, you split it into Students, Courses, and Enrollments tables, then connect them with keys. This reduces redundancy and makes updates safe.

  • Students table: student_id (PK), name, email
  • Courses table: course_id (PK), title, credits
  • Enrollments table: enrollment_id (PK), student_id (FK), course_id (FK)

When you can explain why each table exists and how it connects, you have the normalization mindset C170 is designed to build.

MySQL workflow you should practice

The catalog explicitly mentions MySQL, so practice in a MySQL environment. The goal is not just knowing syntax, but being able to build and test schema changes quickly.

  • Create a new database and tables from scratch
  • Define primary and foreign key constraints
  • Add indexes and verify they are used properly
  • Create and query views for reusable logic

Quality checks before you submit

Before any assessment, run a fast quality checklist:

  • Do table names and columns match the prompt?
  • Are keys and constraints defined correctly?
  • Do joins return the expected number of rows?
  • Do query results match the business rule?

Practice strategy that actually works

The fastest way to improve SQL is to practice on a small, realistic schema. Build a dataset with 3-5 tables (for example, Students, Courses, Enrollments, Instructors, Grades) and write 20-30 queries that answer real questions.

  1. Write the question first (for example: "Which students are enrolled in more than 3 courses?")
  2. Write the SQL that answers it
  3. Check your output against the requirement
  4. Refactor for readability and correctness

Daily 45-minute routine

If you're working full-time, a short daily routine beats a long weekend cram. Use this pattern:

  • 15 minutes: Review a concept (joins, GROUP BY, or normalization)
  • 20 minutes: Write 5-8 practice queries
  • 10 minutes: Check results and fix mistakes

Consistency builds speed. After a week, the same queries will take you half the time.

Debugging and validation checklist

If a query does not work, don't guess. Use a repeatable process:

  • Run the query without filters to inspect raw rows
  • Validate each join with a small subset of data
  • Check for NULL values that break comparisons
  • Verify GROUP BY fields align with SELECT columns

Assessment approach

C170 typically evaluates both conceptual knowledge and SQL performance. Build your prep around mapping each requirement to a query or design choice. If the assessment includes applied tasks, treat every rubric bullet as a deliverable you can test inside your database.

A good habit is to create a checklist with the requirement on the left and the exact query or table change on the right. When every requirement has a matching SQL script, you reduce the risk of missing a key detail under time pressure.

Study plan (2 to 4 weeks)

  1. Week 1: Review SQL basics, SELECT queries, and simple filters
  2. Week 2: Focus on joins, aggregations, and subqueries
  3. Week 3: Data modeling, normalization, and schema design practice
  4. Week 4 (optional): Mixed practice sets and targeted review

If you have prior SQL experience, compress the plan into two weeks and spend extra time on modeling and normalization.

If you only have one week

Short on time? Use a focused sprint:

  1. Days 1-2: SQL fundamentals + 20 practice queries
  2. Days 3-4: Joins + aggregations + subqueries
  3. Day 5: Normalization review + schema design practice
  4. Days 6-7: Full mock run and review weak areas

Exam-day strategy

  • Read each requirement twice and map it to a query or table change
  • Save your work frequently and verify outputs as you go
  • Leave time for a final validation pass across all deliverables

Ready-to-test checklist

  • I can write multi-table joins without guessing
  • I can normalize a table structure to 3NF
  • I can explain why a key or constraint is required
  • I can solve 20+ SQL prompts in a single session

Common mistakes to avoid

  • Memorizing syntax without writing real queries
  • Skipping ER diagram and normalization practice
  • Using the wrong join key or direction
  • Not validating outputs against requirements
