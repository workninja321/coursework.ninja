WGU C195 Software II: Build the JavaFX Scheduling App (JDBC + MySQL) Without Getting Stuck

January 3, 2026 11 min read Course Help

WGU C195 Software II (Advanced Java and Database Integration) is the course where everything you learned in Software I gets upgraded: instead of a purely in-memory JavaFX app, you build a real scheduling system connected to a MySQL database via JDBC.

That jump in complexity is exactly why C195 feels intimidating. The good news: the requirements are consistent, repeatable, and very “rubric-first.” If you build the project in the right order, C195 becomes a sequence of small wins.

This guide gives you a practical path through the project: what to build, which parts to tackle first, what usually causes revisions, and how to handle the “big three” technical pain points: database CRUD, time zones, and validation rules.

Fast-track mindset: Treat C195 like a production-style assignment. Start with the checklist (Task Overview + rubric), then implement requirements in a deliberate order. Random coding is the fastest route to rewrites.

What WGU C195 Is (and What It Usually Isn’t)

C195 is commonly delivered as a performance assessment where you build an application and submit your project. Always confirm your exact assessment type in your course dashboard, but the public requirements for the classic C195 task are widely documented in student resources.

At a high level, you build a desktop scheduling application for a fictional global consulting organization. Your app pulls data from a provided MySQL database, and you cannot modify the database structure. The project typically includes:

  • Login + localization: Translate UI / error messaging for at least two languages.
  • Customer management: Create, update, delete customer records tied to addresses/locations.
  • Appointment scheduling: CRUD appointments linked to customers, with type, time, and contact.
  • Calendar views: Weekly and monthly appointment views.
  • Time zones: Store consistently (often UTC) and display in local time; handle DST.
  • Validation + exception controls: Prevent overlaps, invalid customers, invalid login, and out-of-hours scheduling.
  • Reminders: Alert users on login if they have an appointment within 15 minutes.
  • Reports: Provide multiple database-driven reports.
  • Login activity logging: Append timestamps to a text file.

If you want to see a clear, public listing of these requirements, this GitHub repo includes the full task breakdown as a reference: JasonETrowbridge/WGU-C195.

Academic integrity: Use public repos only to understand requirements and patterns. Do not copy code. Your submission must be your original work and must meet WGU originality requirements.

Prerequisites That Make C195 Easier

You do not need to be a “Java wizard,” but you should be comfortable with the basics. If you struggled in Software I, revisit that foundation first. Start here:

  • WGU C482 Software I: Java Fundamentals and OOP Guide (your direct prerequisite)
  • WGU D191 Advanced Data Management (helpful if SQL/JOINs feel fuzzy)
  • WGU Performance Assessments: How to Pass PAs (rubric-first workflow)

Also: understand how WGU evaluators use rubrics. Even when your app “works,” missing a rubric item can trigger revisions. WGU’s scoring policy for performance assessments is documented here: Performance Assessment Scoring Policy and Rubrics (WGU Knowledge Center).

C195 Project Requirements (Translated into a Build Plan)

Most students get stuck because they treat C195 as “one big project.” Instead, treat it as three subsystems that you connect:

  1. UI subsystem (JavaFX): Scenes, controllers, navigation, validation messages.
  2. Data subsystem (MySQL + JDBC): Connection management, queries, CRUD methods.
  3. Time subsystem (ZonedDateTime strategy): Conversion rules, business-hours checks, upcoming alert logic.

When these are cleanly separated, everything else (reports, calendar views, reminders) becomes much simpler.

Rubric Breakdown (A–J): What Evaluators Typically Expect

If you want C195 to feel predictable, stop thinking “features” and start thinking “rubric evidence.” One of the most useful public references for the classic C195 task is a repo that includes the requirement list (A–J) in plain language. It’s a good reminder of what your submission needs to demonstrate: JasonETrowbridge/WGU-C195.

Requirements change over time, so always follow your current Task Overview. But many versions of C195 include expectations like these:

  • A) Login + localization: Determine the user’s location/locale and translate login and error control messages into two languages. This is often done with ResourceBundle and a localized properties file.
  • B) Customer records: Create and maintain customer records in the database (commonly name, address, phone). Watch for foreign key constraints when deleting.
  • C) Lambda expression(s): Use lambda expressions in a meaningful way tied to scheduling/maintaining appointments. Don’t just “add a lambda to satisfy the rubric”—make it simplify code (e.g., filtering lists, event handlers, table row actions) and explain why.
  • D) Calendar views: Provide month and week views describing appointments. Most students implement this as filtered views of the appointment list.
  • E) Time zones + DST: Automatically adjust appointment times based on time zones and daylight saving time. The most reliable approach is “store canonical, display local.”
  • F) Exception controls: Prevent out-of-business-hours scheduling, overlapping appointments, invalid customer data, and incorrect login attempts. Many rubrics also require at least two different mechanisms of exception control (e.g., input validation + exception catching + alerting).
  • G) Alerts/popups: Use lambdas and/or centralized helpers to generate standard popups/alerts. Students often create a reusable alert helper to avoid duplicating 50+ lines of UI code.
  • H) 15-minute reminder: On login, alert if an appointment is within 15 minutes. The easiest way is a query for “now through now+15 minutes” in canonical time, then convert to display time.
  • I) Reports: Provide specific reports (appointment types by month, schedules, and one additional). Treat this as an SQL + presentation task, not a UX design project.
  • J) Login activity logging: Append timestamped logins to a text file (often named something like login_activity.txt). Include failed attempts if your Task Overview requires it.
Don’t miss this hidden “stat”: Some versions of the task explicitly call out originality limits (e.g., no more than 30% closely paraphrased/quoted overall, and no more than 10% matching any single source). Treat public repos as requirements references only, and write your own code.

Architecture That Keeps You Sane (and Helps You Finish Faster)

C195 doesn’t require enterprise architecture, but your future self will thank you if you adopt a small, consistent structure. Successful submissions tend to converge on a simple MVC-ish pattern:

  • model: Plain data objects like Customer, Appointment, Contact (fields + getters/setters).
  • dao (or repository): Database CRUD methods (getAllCustomers, addCustomer, updateAppointment, etc.).
  • controller: JavaFX controllers that load data into tables/forms and call DAO methods.
  • util: Reusable helpers (time conversion, alert dialogs, input parsing, validation).
  • view: FXML files and UI assets.

This package split has two practical benefits:

  1. Debugging is faster: If a record isn’t saving, you know to look at DAO first, not the controller.
  2. Rubric coverage is clearer: You can point to the exact code path that proves each requirement.

A widely shared C195 acceleration tip (from recent Reddit threads) is to create “helper methods” early: a single function for standardized alerts, a single function for converting time zones, and helper methods for mapping IDs to names for combo boxes. The main reason this works: C195 is filled with repeated patterns, and helpers reduce both code and mistakes.

Validation Rules: How to Avoid the Two Biggest Revision Triggers

In most C195 versions, revisions happen because the app allows a user to schedule something the rubric says must be blocked. The two most common blockers are overlaps and business hours.

Overlapping appointments

Overlap logic seems easy until you implement “modify appointment.” A robust approach checks for overlaps using canonical time, and when editing, excludes the appointment currently being edited.

Test overlap scenarios explicitly:

  • Exact same start/end time as an existing appointment
  • New start inside an existing appointment window
  • New end inside an existing appointment window
  • New appointment fully wraps an existing appointment
  • Back-to-back appointments (end == start) — confirm whether this is allowed in your rubric

Business hours

Many rubrics define business hours in a specific time zone (often Eastern), not the student’s local zone. The safest strategy is:

  • Parse input as the user’s local time
  • Convert to the rubric’s business-hours zone
  • Validate the converted time is inside the allowed window
  • Store in canonical time
Reality check: If your app “works on your machine” but fails business-hours checks when evaluated in another locale, you’ll get a revision. Validate in the expected zone, not just your own.

Input validation vs exceptions

When the rubric says “exception controls,” that doesn’t mean your program should crash. It usually means: proactively validate input, and show a clear message when something is invalid. You can satisfy “two different mechanisms” by combining:

  • Form validation (empty fields, invalid IDs, invalid date ranges)
  • Try/catch around database operations (SQL exceptions with a user-friendly alert)
  • Logical checks (overlap and business hour rules)

Make your validation visible

Evaluators can’t read your mind. If you block an invalid appointment, show a clear alert that says what rule was violated and what the allowed range is. This reduces “works but not evident” feedback.

Bonus: The 15-minute alert is easier than it sounds

Students commonly overcomplicate this. You don’t need a background scheduler. Most rubrics only require an alert immediately after login. The practical flow is: query appointments, find any within the next 15 minutes, and display an alert. If none exist, show a “No upcoming appointments” message if your task requires it.

Logging user activity

Logging is usually a single append operation, but don’t leave it to the end. Create a dedicated method to append each login attempt with a timestamp. That way, even if you refactor the login flow, logging stays consistent.

Localization: keep it focused

For localization, you usually don’t need to translate every screen. Most task versions focus on the login form and error messages. Implement it early, then leave it alone—last-minute localization changes are a fast way to introduce bugs.

Reports: use SQL for the heavy lifting

Another recurring tip from successful students: use SQL aggregation functions where possible. A “types by month” report is naturally expressed as GROUP BY month + type. You can still display results in JavaFX, but let the database do the counting.

Javadoc and documentation

Some cohorts require strong documentation. If your course requires Javadocs, write them as you build. It’s far easier than trying to retroactively explain how your own code works the night before submission.

Keep your submission runnable

If you’re using IntelliJ + JavaFX, do at least one “clean import” test before you submit: zip/export, unzip, and run as if you were the evaluator. If dependencies aren’t set up correctly, this is where you catch it—before it becomes a revision.

UI: What Screens You Usually Need

Exact layouts vary, but most passing submissions include at least:

  • Login screen (localized labels/messages, plus login logging).
  • Main dashboard with tabs or sections for Customers and Appointments.
  • Customer form (add/modify in one scene is common).
  • Appointment form (add/modify in one scene is common).
  • Reports view (TableView or simple output is fine if it meets requirements).

Database: CRUD Without Chaos

Your database CRUD strategy is a huge success lever. If you’re comfortable with patterns, use a small DAO layer:

  • DB connection helper: one place that creates the JDBC connection and handles errors.
  • DAO methods: “get all appointments”, “insert appointment”, “update customer”, etc.
  • Model classes: Customer, Appointment, Contact, User (as needed).
Pro tip: Students who finish faster typically create reusable helpers early (alerts, time conversions, and ID lookups). A recent Reddit guide highlights this as a “do this first” move.

One of the most shared “passed fast” posts is here: Software II (C195) Passed in 5 Days (Tips/Guide). Even if you’re not trying to finish in five days, the workflow advice is solid.

Setup and Tools: Avoid the “Day 1 Dependency Trap”

A surprising amount of C195 pain is not “coding,” it’s setup: JavaFX dependencies, MySQL Connector/J, and a project structure that exports cleanly for submission.

Use Maven/Gradle for Dependencies (When Allowed)

Students increasingly recommend using Maven-based JavaFX templates and importing MySQL Connector/J via Maven instead of manually managing jar files. The “Passed in 5 Days” Reddit post specifically calls out using mysql-connector-j (new naming) and avoiding outdated drivers.

Test the Export Early

Do not wait until submission day to export/zip and test on a clean project import. Many “my evaluator couldn’t run it” revisions are packaging problems, not code problems.

JDBC + MySQL: The Minimum You Must Get Right

At minimum, your database layer should support:

  • Read lists for customers, appointments, contacts, etc.
  • Create/update/delete customers.
  • Create/update/delete appointments (with overlap and business-hours checks).
  • Report queries that aggregate data (counts per month, schedules per consultant/contact, and a third custom report).

Keep your SQL simple and safe:

  • Prefer prepared statements.
  • Log or display errors with meaningful messages (and keep it user-friendly).
  • When doing reports, it’s usually easier to let SQL do the grouping (e.g., COUNT, GROUP BY) instead of doing everything in Java loops.

Time Zones and Business Hours: The Part That Breaks Apps

Time zones are the reason working apps get rejected. You can avoid 90% of these issues with one consistent rule:

Rule: Convert inputs into a single canonical zone for storage (often UTC), and convert back to the user’s local time for display.

Then design validations around that rule:

  • Overlap checks should compare canonical times (UTC or a consistent zone) to avoid local-time ambiguity.
  • Business hours checks should explicitly convert to the required time zone (commonly Eastern). Do not assume business hours are in your local time—confirm the expected window in your task overview.
  • DST handling: Always use Java’s modern time API (e.g., ZonedDateTime) instead of older Date/Calendar logic.

If you’re stuck on converting between zones, the Reddit guide explicitly recommends using withZoneSameInstant to keep the moment-in-time consistent while changing zones.

Reports: Keep It Simple, Keep It Correct

C195 typically requires multiple reports (for example, appointment counts by month, a schedule for each consultant/contact, and one additional report of your choice). The goal is not a “pretty dashboard.” The goal is: prove you can query and present data.

Strong report strategies:

  • Write one SQL query per report and display results in a TableView.
  • Use SQL functions like MONTH(), COUNT(), and GROUP BY where appropriate.
  • Keep report UI minimal and predictable so you don’t burn time on layouts.

A Step-by-Step C195 Build Order (That Minimizes Rework)

If you implement in this order, you reduce the chance that you build UI against the wrong data model or rewrite time logic later:

  1. Project setup: Confirm JavaFX runs. Confirm DB connection works. Create a place for config/credentials (do not hardcode secrets).
  2. Models: Create model classes for Customer, Appointment, Contact, and any supporting data.
  3. DB helper + DAOs: Create reusable methods for connecting and for the CRUD operations you know you will need.
  4. Login: Implement login UI + logging + localization early so it’s not forgotten at the end.
  5. Customer CRUD: Build customer screen and forms. Validate input carefully.
  6. Appointment CRUD: Build appointment screen and forms. Add time zone conversions and validations (business hours + overlap).
  7. 15-minute alert: Run the upcoming appointment query right after login.
  8. Calendar views: Implement month/week views by filtering the appointment list.
  9. Reports: Add the required report queries and display.
  10. Final pass: Walk the rubric line-by-line. Record a short “demo checklist” for yourself to verify every path works.

Common C195 Pitfalls (and How to Avoid Them)

1) Time zone conversions that “look right” but fail edge cases

  • Choose a canonical storage zone (commonly UTC) and stick to it.
  • Convert to local time only for display.
  • Test with appointments near midnight and around DST boundaries.

2) Overlap checks that are incomplete

  • Check overlap using the exact logic described in your task overview.
  • Run overlap checks against the database or a canonical in-memory list.
  • Remember to handle updates (exclude the appointment being edited).

3) “It works” but the rubric item is missing

  • Missing report requirements
  • Missing login activity logging
  • Missing “appointment within 15 minutes” alert
  • Missing required exception controls or missing the “two different mechanisms” requirement
Checklist habit: Convert the Task Overview page into a checklist and only check items off after you’ve tested them in the running app. This is repeatedly recommended by students who finish quickly.

Submission Checklist: What to Verify Before You Upload

Before submission, run a clean demo path with your rubric open. Confirm:

  • You can log in, and logins append to the activity file.
  • Localization changes messages/labels as required.
  • Customer add/modify/delete works and updates the database.
  • Appointment add/modify/delete works and updates the database.
  • Business-hours and overlap validations prevent invalid scheduling.
  • The 15-minute upcoming appointment alert triggers (test by creating an appointment close to now).
  • Calendar views (month/week) correctly show appointments.
  • All required reports generate correct output.
  • The exported project runs on a clean machine/import (dependency check).
